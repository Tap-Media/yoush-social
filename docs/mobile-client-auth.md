# Mobile Client Auth Integration

This document describes how a mobile client should integrate registration and sign-in against:

`https://dev.yoush.social.tapofthink.com`

It covers the Mastodon-compatible auth APIs exposed by this server.

## Scope

Use this document for the following flows:

- Create or provision an OAuth application
- Register a new account
- Sign in an existing account with OAuth authorization code
- Exchange authorization code for an access token
- Verify the signed-in account

Do not use the `Internal Auth APIs` section from `API_LIST.md` for this base URL. That section appears to describe a different service.

## Important Constraints

- This server does **not** support OAuth password grant. Do not call `/oauth/token` with `grant_type=password`.
- The supported OAuth grant flows are `authorization_code` and `client_credentials`.
- PKCE with `S256` is supported and should be used by mobile clients.
- Applications created through `/api/v1/apps` are confidential by default, so the response includes both `client_id` and `client_secret`.
- Successful sign-up returns an access token, but the new user may still be unconfirmed or pending approval.

## Endpoints Summary

| Purpose | Method | Path |
|---|---|---|
| Create OAuth app | `POST` | `/api/v1/apps` |
| Get app token | `POST` | `/oauth/token` |
| Register account | `POST` | `/api/v1/accounts` |
| Start OAuth login | `GET` | `/oauth/authorize` |
| Exchange auth code | `POST` | `/oauth/token` |
| Verify current user | `GET` | `/api/v1/accounts/verify_credentials` |
| Revoke token | `POST` | `/oauth/revoke` |
| Resend confirmation email | `POST` | `/api/v1/emails/confirmations` |
| Check email confirmed | `GET` | `/api/v1/emails/check_confirmation` |

## `APP_NAME` And `REDIRECT_URI`

### `APP_NAME`

`APP_NAME` is the human-readable name of the mobile application.

It is sent to the server as `client_name` when calling `POST /api/v1/apps`.

What it is used for:

- Stored as the OAuth application name on the server
- Displayed to the user on the OAuth consent screen
- Helps the user recognize which app is requesting access

Where the value comes from:

- It is **not** defined by this server repo for the mobile client
- It should come from the mobile app configuration, build constants, or remote config
- It should be chosen by the mobile team and stay stable per app and environment

Recommended examples:

- `Yoush Mobile Dev`
- `Yoush Mobile iOS Dev`
- `Yoush Mobile Android Dev`

### `REDIRECT_URI`

`REDIRECT_URI` is the OAuth callback URI that the server redirects back to after the user logs in and approves access.

What it is used for:

- Passed when creating the OAuth app
- Passed again when opening `/oauth/authorize`
- Passed again when exchanging the authorization code at `/oauth/token`
- Used by the mobile app to receive the `code` and `state`

Where the value comes from:

- It is **not** generated by this server
- It must come from the mobile client's deep link or universal link configuration
- It must exactly match across all OAuth steps

Typical sources by platform:

- iOS: `Info.plist` URL Types or Associated Domains
- Android: `intent-filter` or App Links configuration

Recommended examples:

- Custom scheme: `yoush-dev://oauth/callback`
- Universal link: `https://mobile-dev.yoush.social/oauth/callback`

If the mobile team has not chosen a value yet, the value must be defined first in the mobile app before OAuth login can work.

## Credentials And Where They Come From

| Value | Source |
|---|---|
| `APP_NAME` | Chosen by mobile team, stored in mobile app config |
| `REDIRECT_URI` | Chosen by mobile team, stored in deep link config |
| `client_id` | Returned by `POST /api/v1/apps` |
| `client_secret` | Returned by `POST /api/v1/apps` |
| `app access token` | Returned by `POST /oauth/token` with `grant_type=client_credentials` |
| `user access token` | Returned by sign-up or authorization code exchange |

## Recommended Provisioning Model

There are two workable approaches for mobile:

### Option A: Pre-provision one app per environment

Create one OAuth application for each environment and platform, then ship these values in app config or remote config:

- `APP_NAME`
- `REDIRECT_URI`
- `client_id`
- `client_secret`

This is the simplest model for production mobile apps.

### Option B: Create an OAuth app from the mobile client

The mobile app can call `POST /api/v1/apps` on first launch and persist the returned `client_id` and `client_secret` locally.

This works, but it creates a new OAuth application record per install or per device.

## Recommended Scopes

For auth-only flows, start with:

- `read write`

For a full mobile social client, use:

- `read write follow push`

## Flow 1: Register A New Account

Registration requires an **app token**, not a user token.

### Step 1: Create Or Provision An OAuth App

Request:

```http
POST /api/v1/apps
Content-Type: application/x-www-form-urlencoded

client_name=Yoush%20Mobile%20Dev
redirect_uris=yoush-dev%3A%2F%2Foauth%2Fcallback
scopes=read%20write
website=
```

Successful response example:

```json
{
  "id": "123",
  "name": "Yoush Mobile Dev",
  "client_id": "app_uid",
  "client_secret": "app_secret",
  "client_secret_expires_at": 0,
  "redirect_uris": ["yoush-dev://oauth/callback"],
  "redirect_uri": "yoush-dev://oauth/callback",
  "scopes": ["read", "write"],
  "website": null,
  "vapid_key": "..."
}
```

Persist these values for later use:

- `client_id`
- `client_secret`
- `redirect_uri`

### Step 2: Get An App Token

Request:

```http
POST /oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
client_id=app_uid
client_secret=app_secret
scope=read%20write
```

Successful response example:

```json
{
  "access_token": "app_access_token",
  "token_type": "Bearer",
  "scope": "read write",
  "created_at": 1713512345
}
```

### Step 3: Register The User

Request:

```http
POST /api/v1/accounts
Authorization: Bearer app_access_token
Content-Type: application/x-www-form-urlencoded

username=alice
email=alice@example.com
password=12345678
agreement=true
date_of_birth=2000-01-01
locale=vi
time_zone=Asia/Ho_Chi_Minh
invite_code=
reason=
```

Supported payload fields:

| Field | Required | Notes |
|---|---|---|
| `username` | Yes | Account username |
| `email` | Yes | User email |
| `password` | Yes | Password |
| `agreement` | Yes in practice | Send `true` |
| `date_of_birth` | Conditional | Needed when minimum age is enabled |
| `locale` | No | Example: `vi` |
| `time_zone` | No | Example: `Asia/Ho_Chi_Minh` |
| `invite_code` | No | Required if registrations are invite-only |
| `reason` | No | Used as invite request text |

Successful response example:

```json
{
  "access_token": "user_access_token",
  "token_type": "Bearer",
  "scope": "read write",
  "created_at": 1713512345
}
```

### Step 4: Handle Post-Sign-Up State

The sign-up is successful, but the account may still not be fully usable.

Possible account states immediately after sign-up:

- Email not confirmed yet
- Pending moderator approval

Typical API errors after sign-up:

```json
{ "error": "Your login is missing a confirmed e-mail address" }
```

```json
{ "error": "Your login is currently pending approval" }
```

If the account is awaiting email confirmation, the mobile client can call:

```http
POST /api/v1/emails/confirmations
Authorization: Bearer user_access_token
Content-Type: application/x-www-form-urlencoded

email=alice@example.com
```

This endpoint is only available to the same application that created the account.

The mobile client can also check whether the email is confirmed:

```http
GET /api/v1/emails/check_confirmation
Authorization: Bearer user_access_token
```

Response example:

```json
true
```

## Flow 2: Sign In An Existing Account

For existing users, use OAuth authorization code flow with PKCE.

### Step 1: Ensure The App Credentials Exist

Before login starts, the app must already have:

- `client_id`
- `client_secret`
- `REDIRECT_URI`
- requested scopes

### Step 2: Generate PKCE Values

The mobile client should generate:

- `code_verifier`
- `code_challenge = BASE64URL(SHA256(code_verifier))`
- `state`

Use:

- `code_challenge_method=S256`

Do not use `plain`.

### Step 3: Open The Authorization URL In The Browser

Open this URL in the system browser or platform OAuth session:

```text
https://dev.yoush.social.tapofthink.com/oauth/authorize
  ?client_id=app_uid
  &response_type=code
  &redirect_uri=yoush-dev%3A%2F%2Foauth%2Fcallback
  &scope=read%20write%20follow%20push
  &state=random_state
  &code_challenge=generated_code_challenge
  &code_challenge_method=S256
```

What happens next:

- The server shows the hosted sign-in page
- The user signs in there
- If the account uses OTP or WebAuthn, the server handles the challenge pages there
- The server shows the OAuth consent page
- After approval, the server redirects to `REDIRECT_URI?code=...&state=...`

### Step 4: Parse The Redirect In The Mobile App

The app must listen for the redirect URI and extract:

- `code`
- `state`

The app should verify that the returned `state` matches the original request.

### Step 5: Exchange The Authorization Code For A User Token

Request:

```http
POST /oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
client_id=app_uid
client_secret=app_secret
redirect_uri=yoush-dev%3A%2F%2Foauth%2Fcallback
code=authorization_code
code_verifier=original_code_verifier
```

Successful response example:

```json
{
  "access_token": "user_access_token",
  "token_type": "Bearer",
  "scope": "read write follow push",
  "created_at": 1713512345
}
```

### Step 6: Verify The Signed-In User

Request:

```http
GET /api/v1/accounts/verify_credentials
Authorization: Bearer user_access_token
```

Successful response example:

```json
{
  "id": "42",
  "username": "alice",
  "acct": "alice",
  "display_name": "Alice",
  "locked": false,
  "bot": false,
  "discoverable": true,
  "created_at": "2026-04-20T10:00:00.000Z"
}
```

## Common Errors

### Registration Closed

```json
{ "error": "Forbidden" }
```

Typical reasons:

- Registrations are closed
- Single-user mode is enabled
- OAuth-only login is enabled
- Sign-up IP is blocked

### Wrong Token Type For Sign-Up

```json
{ "error": "This method requires an client credentials authentication" }
```

Cause:

- `/api/v1/accounts` was called with a user token instead of an app token

### Validation Error During Sign-Up

```json
{
  "error": "Validation failed: Username can't be blank",
  "details": {
    "username": [
      {
        "error": "ERR_BLANK",
        "description": "can't be blank"
      }
    ]
  }
}
```

### Unconfirmed Or Pending User

```json
{ "error": "Your login is missing a confirmed e-mail address" }
```

```json
{ "error": "Your login is currently pending approval" }
```

## What The Mobile Client Should Persist

Persist these values securely:

- `client_id`
- `client_secret`
- `REDIRECT_URI`
- selected scopes
- `user_access_token`
- token issue time

If the app supports log out, also call:

```http
POST /oauth/revoke
Content-Type: application/x-www-form-urlencoded

client_id=app_uid
client_secret=app_secret
token=user_access_token
```

## Implementation Notes For Mobile Teams

- Use browser-based OAuth sessions instead of embedding the login page in a raw WebView.
- Always keep `REDIRECT_URI` identical across app registration, authorization request, and token exchange.
- Always send `state` and verify it on return.
- Use PKCE `S256` for every authorization code login.
- Do not assume sign-up success means the account is immediately active.
- If the mobile team has not yet defined `APP_NAME` and `REDIRECT_URI`, OAuth login cannot be finalized until those two values are agreed and configured in the mobile app.
